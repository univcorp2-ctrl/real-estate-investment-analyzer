# Kantei-san 要件定義書

## 0. ドキュメントの目的

このドキュメントは、収益不動産投資を初期判定するためのロジックを **Kantei-san** と名付け、初心者でも理解できるように要件定義として整理したものです。

Kantei-san は、物件を見たときに次のような疑問に答えるための仕組みです。

- この物件は毎月お金が残るのか
- 融資返済に耐えられるのか
- 表面利回りだけで良く見えているだけではないか
- 土地値や出口戦略に安全性があるのか
- 価格は高すぎないか
- 災害や人口減少などのリスクはないか
- 初心者が見ても危険な物件を避けられるか

## 1. Kantei-san の考え方

### 1.1 不動産投資で最も危険な見方

不動産投資の初心者がやりがちな危険な見方は、次のようなものです。

```text
表面利回りが高いから良い物件だ
価格が安いから良い物件だ
新築だから安全だ
RCだから安全だ
駅に近いから安全だ
満室だから大丈夫だ
```

これらは一部だけを見る判断です。実際には、以下を同時に見ないと危険です。

- 賃料収入
- 空室リスク
- 管理費、修繕費、税金、保険料
- 融資返済額
- 金利上昇
- 売却時の出口
- 土地値
- 建物劣化
- 周辺人口
- ハザード
- 再建築可否
- 価格の妥当性

Kantei-san は、これらをなるべく同じ画面で比較できるようにするロジックです。

### 1.2 Kantei-san の役割

Kantei-san は、最終的な投資判断ツールではなく、初期スクリーニングツールです。

役割:

1. 大量の物件から候補を絞る
2. 危険な物件を早めに弾く
3. 物件ごとの確認ポイントを明確にする
4. 融資前提を変えたときの影響を見る
5. 価格交渉の根拠を作る
6. 土地値・出口・リスクを含めた見方にする

使ってはいけない使い方:

- S判定だから必ず買う
- A判定だから現地確認なしで買う
- C判定だから絶対に買わない
- 税金、修繕、法務、現地調査を省略する
- 銀行の本審査を通さずに安全だと判断する

## 2. システム全体像

Kantei-san は、次の流れで判定します。

```text
物件情報を入力
  ↓
収支指標を計算
  ↓
公的データ・相場データを取得
  ↓
土地値・価格妥当性を計算
  ↓
6軸で採点
  ↓
重大リスクをチェック
  ↓
S/A/B/C/D/E の総合判定を出す
  ↓
確認ポイントを表示する
```

## 3. 対象ユーザー

### 3.1 メインユーザー

- 不動産投資初心者
- 副業で不動産投資を検討している個人
- 収益物件を比較したい投資家
- 物件を大量に見る仲介・買取・再販担当者
- 融資条件を変えてシミュレーションしたい人

### 3.2 初心者向けに必要な工夫

初心者には、単語だけを出しても意味が伝わりません。

そのため、画面やレポートでは次のように表示します。

悪い例:

```text
DSCR: 0.92
CCR: -3.5%
土地値カバー率: 21%
```

良い例:

```text
DSCR: 0.92
意味: 物件の収入だけではローン返済を賄えない可能性があります。
判断: 危険。返済条件の見直し、価格交渉、自己資金増加が必要です。
```

## 4. 入力項目

### 4.1 最低限必要な入力

最低限、以下がないと投資判断の精度が大きく落ちます。

| 項目 | 必須 | 単位 | 説明 |
|---|---|---:|---|
| 物件価格 | 必須 | 万円 | 買う価格。利回り、LTV、価格妥当性の基準 |
| 月額賃料 | 必須 | 万円/月 | 収入のもと。現況賃料か想定賃料かを分ける |
| 構造 | 必須 | 文字列 | 木造、鉄骨、RC、SRCなど。耐用年数と修繕リスクに影響 |
| 築年数 | 必須 | 年 | 建物の劣化、融資期間、出口に影響 |
| 借入額 | 必須 | 万円 | ローン元本。返済額、LTV、CFに影響 |
| 金利 | 必須 | %/年 | 返済額に影響 |
| 返済期間 | 必須 | 年 | 月々の返済額に影響 |

### 4.2 強く推奨する入力

| 項目 | 単位 | なぜ必要か |
|---|---:|---|
| 所在地 | 住所 | 地価、ハザード、人口、駅距離を取得するキー |
| 緯度経度 | 度 | API・GISデータ取得のキー |
| 土地面積 | m2 | 土地値カバー率に必須 |
| 建物面積 | m2 | 積算価格、修繕費、類似比較に必要 |
| 固定資産税 | 円/年 | 実際のNOIに大きく影響 |
| 管理費 | 円/月または円/年 | 実際のNOIに影響 |
| 修繕費 | 円/年 | 築古物件では特に重要 |
| 保険料 | 円/年 | NOIに影響 |
| 路線価 | 円/m2 | 土地値カバー率の主要データ |
| 近隣賃料 | 円/月 | 賃料が妥当か確認するため |
| 近隣売買事例 | 円 | 価格妥当性を見るため |
| ハザード情報 | 該当/非該当 | 融資、保険、出口に影響 |

### 4.3 URLパラメータとして想定する項目

既存のサンプルURLから、次のパラメータをKantei-sanでも扱います。

| パラメータ | 意味 | 単位 | 例 |
|---|---|---:|---|
| `caseId` | 案件ID | 文字列 | RE-20260406-12445020 |
| `price` | 物件価格 | 万円 | 880 |
| `str` | 構造 | 文字列 | RC |
| `age` | 築年数 | 年 | 0 |
| `loanAmt` | 借入額 | 万円 | 880 |
| `rent` | 月額賃料 | 万円/月 | 5.8 |
| `br` | 推定月額賃料 | 千円/月 | 58 |
| `ir` | 金利 | %/年 | 2 |
| `ly` | 返済期間 | 年 | 30 |
| `name` | 物件名 | 文字列 | 東村山市 戸建て |
| `ref` | 流入元 | 文字列 | portfolio |

`rent=0` かつ `br=58` のような場合は、`br=58` を 58,000円/月、つまり 5.8万円/月として扱います。

## 5. 収支計算

### 5.1 表面利回り

表面利回りは、最も基本的な指標です。

```text
表面利回り = 年間満室賃料 ÷ 物件価格 × 100
```

例:

```text
物件価格 = 880万円
月額賃料 = 5.8万円
年間賃料 = 5.8万円 × 12 = 69.6万円
表面利回り = 69.6万円 ÷ 880万円 × 100 = 7.91%
```

初心者向け説明:

- 表面利回りは、費用を引く前のざっくりした利回りです。
- 管理費、修繕費、税金、保険、空室、ローン返済はまだ引いていません。
- 表面利回りだけで投資判断をしてはいけません。

### 5.2 実効総収入

空室を考慮した収入です。

```text
実効総収入 = 年間満室賃料 × (1 - 空室損率)
```

空室損率5%なら、満室賃料の95%を収入として見ます。

### 5.3 NOI

NOIは、不動産の営業利益のようなものです。

```text
NOI = 実効総収入 - 運営費 - 修繕予備費 - 固定資産税 - 保険料
```

MVPでは簡易式として以下を使います。

```text
NOI = 年間賃料 × (1 - 空室損率) - 年間賃料 × 運営費率 - 物件価格 × 修繕予備率
```

デフォルト:

| 項目 | 初期値 |
|---|---:|
| 空室損率 | 5% |
| 運営費率 | 25% |
| 修繕予備率 | 物件価格の0.5%/年 |

初心者向け説明:

- NOIはローン返済前の利益です。
- 物件そのものが稼ぐ力を見る数字です。
- ローン条件に関係なく、物件の収益力を比較できます。

### 5.4 年間返済額

ローン返済額は、元利均等返済を基本とします。

```text
月返済額 = 借入元本 × 月利 × (1 + 月利)^返済回数 ÷ ((1 + 月利)^返済回数 - 1)
年間返済額 = 月返済額 × 12
```

初心者向け説明:

- 金利が高いほど返済額は増えます。
- 返済期間が短いほど月々の返済額は増えます。
- 借入額が大きいほど返済額は増えます。
- 表面利回りが高くても、返済額が大きければ毎月赤字になります。

### 5.5 DSCR

DSCRは返済安全性を見る指標です。

```text
DSCR = NOI ÷ 年間返済額
```

意味:

| DSCR | 意味 |
|---:|---|
| 1.5以上 | かなり余裕がある |
| 1.3以上 | 安全域がある |
| 1.15以上 | 最低限の余裕がある |
| 1.0以上 | 返済は一応賄える |
| 1.0未満 | 物件収入だけでは返済できない可能性が高い |

初心者向け説明:

- DSCRが1.0未満は危険です。
- たとえばDSCR 0.9なら、ローン返済に対してNOIが90%しかありません。
- 足りない分は自己資金や給与から補填する必要があります。

### 5.6 月間キャッシュフロー

毎月残るお金です。

```text
月間CF = (NOI - 年間返済額) ÷ 12
```

初心者向け説明:

- 月間CFがプラスなら、ローン返済後にお金が残ります。
- 月間CFがマイナスなら、毎月持ち出しです。
- ただし、月間CFが少しプラスでも、大規模修繕で一気に赤字になる可能性があります。

### 5.7 CCR

CCRは自己資金に対する利回りです。

```text
CCR = 年間税前CF ÷ 自己資金投入額 × 100
```

```text
自己資金投入額 = 物件価格 - 借入額 + 購入諸費用
```

初心者向け説明:

- CCRは、実際に自分が出した現金がどのくらい働いているかを見る指標です。
- フルローンに近いほど自己資金が少なくなり、CCRが大きく見えることがあります。
- ただし、自己資金が少ない投資は安全とは限りません。

## 6. 6軸評価

Kantei-san は、各5点、合計30点で評価します。

| 軸 | 何を見るか | 初心者向けの意味 |
|---|---|---|
| 収益性 | 利回り、CF、NOI | ちゃんと儲かりそうか |
| 立地 | 駅、人口、流動性 | 借り手・買い手がいそうか |
| 融資条件 | DSCR、金利、LTV | ローン返済に無理がないか |
| リスク耐性 | 空室、家賃下落、災害 | 悪いことが起きても耐えられるか |
| 価格妥当性 | 相場、収益還元、積算 | 高値掴みではないか |
| 出口戦略 | 土地値、残存耐用年数、売却性 | 最後に売れるか、逃げ道があるか |

## 7. 総合ランク

### 7.1 点数による基本ランク

| 総点 | ランク | 意味 |
|---:|---|---|
| 22〜30 | S | 積極検討。ただし最終確認は必須 |
| 20〜21 | A | 有力候補。確認ポイントを潰せば検討可能 |
| 15〜19 | B | 条件次第。価格交渉や融資条件次第 |
| 12〜14 | C | 慎重検討。初心者は避ける方が無難 |
| 8〜11 | D | 原則見送り。危険要素が多い |
| 0〜7 | E | 見送り。投資対象として不適合の可能性が高い |

### 7.2 ゲート判定

総点が高くても、重大リスクがある場合は上位ランクにしません。

例:

```text
総点24点でも、月間CFが赤字ならSにはしない
総点22点でも、DSCRが1.0未満ならD以下に落とす
総点20点でも、土地値が低く出口が厳しいならA止まりにする
```

ゲート判定の目的:

- 見た目の点数に騙されないため
- 重大リスクを強制的に反映するため
- 初心者が危険物件を過大評価しないようにするため

Kantei-san の画面・API・レポートでは、以下を出力します。

### 8.1 サマリー

| 出力 | 説明 |

### 8.2 収支指標

| 出力 | 説明 |
|---|---|
| 表面利回り | 費用控除前の利回り |
| NOI | ローン返済前の営業利益 |
| 年間返済額 | ローン返済額 |
| DSCR | 返済安全性 |
| 月間CF | 毎月の手残り |
| CCR | 自己資金に対する利回り |
| LTV | 物件価格に対する借入割合 |

### 8.3 評価内訳

| 出力 | 説明 |

| 6軸点数 | 収益性、立地、融資、リスク、価格、出口 |

### 8.4 データ品質

| 出力 | 説明 |

## 9. MVPと本番版の違い

### 9.1 MVP

MVPでは、まず以下を実装します。

- URLパラメータ・フォーム入力
- 収支計算
- 6軸スコア
- ランク判定
- 国交省APIからの公的データ取得
- 土地値カバー率の簡易計算
- 価格妥当性の簡易計算
- Cloud RunでのAPI稼働

### 9.2 本番版

本番版では、以下を追加します。

- 住所ジオコーディング
- 路線価CSV/商用API連携
- 固定資産税評価額の管理
- 近隣賃料DB
- 管理費・修繕費の実績DB
- ハザード詳細判定
- 物件タイプ別ロジック
- 税引後CF
- 減価償却
- 譲渡税
- 融資条件の複数シナリオ比較
- レポートPDF出力
- 物件履歴管理

## 10. 非機能要件

### 10.1 セキュリティ

- APIキーはGitHubに入れない
- APIキーはCloud Secret Managerで管理する
- ブラウザにAPIキーを渡さない
- サーバー側で外部APIを呼ぶ
- 入力値はバリデーションする
- ログに個人情報やAPIキーを出さない

### 10.2 可用性

- 外部APIが落ちても、手入力値だけで最低限の判定ができる
- API取得エラーを画面に表示する
- 取得済みデータをキャッシュできる設計にする

### 10.3 保守性

- しきい値はコードにベタ書きしすぎず、将来的に設定ファイル化する
- 物件タイプ別にスコアリングを分けられるようにする
- データソースを差し替えられるようにする

### 10.4 監査性

- どの入力で、どのスコアになったか保存できる
- どのデータソースを使ったか保存できる
- 推定値と実測値を区別できる
- 判定日時を保存する

## 11. Kantei-san の重要な思想

Kantei-san は、単に「利回りが高い物件」を探すツールではありません。

目指すのは、次のような判断です。

```text
利回りは高いが、出口が弱いので初心者には危険
利回りは普通だが、土地値が厚く、DSCRも安全なので検討価値あり
CFは良いが、ハザードと人口減少が強いため価格交渉が必要
総点は高いが、データ不足が多いのでA止まり
```

つまり、Kantei-san は「儲かりそうか」だけでなく、「失敗しにくいか」を重視します。
